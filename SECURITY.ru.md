# Описание модели безопасности ZFS

## 1. Обзор

ZFS реализует многоуровневую модель защиты данных, охватывающую контроль целостности, шифрование, разграничение доступа и аудит операций.

## 2. Контроль целостности данных

### 2.1 Механизм контрольных сумм

Каждый блок данных в ZFS снабжается контрольной суммой. При чтении сумма пересчитывается и сравнивается с сохранённой — повреждённый блок обнаруживается немедленно.

Поддерживаемые алгоритмы:

| Алгоритм | Применение |
|----------|------------|
| SHA-256 | Рекомендован для данных повышенной важности |
| BLAKE3 | Высокопроизводительный, рекомендован по умолчанию |
| Fletcher-4 | Используется для метаданных (быстрый) |
| SHA-512/256 | Альтернатива SHA-256 |

### 2.2 Самовосстановление

При использовании RAID-Z ZFS автоматически восстанавливает повреждённые блоки из избыточных копий. Операция `zpool scrub` проверяет весь пул и исправляет обнаруженные ошибки.

### 2.3 Защита метаданных

Метаданные файловой системы (дерево блоков) хранятся в нескольких копиях и также защищены контрольными суммами. Повреждение метаданных не приводит к потере пула.

## 3. Шифрование

### 3.1 Нативное шифрование датасетов

ZFS поддерживает шифрование на уровне датасета без использования сторонних инструментов.

| Параметр | Значение |
|----------|----------|
| Алгоритмы | AES-256-GCM, AES-256-CCM |
| Управление ключами | Встроенное (keystore), внешнее через PKCS#11 |
| Форматы ключей | Парольная фраза, hex-ключ, raw-ключ |
| Шифрование метаданных | Имена файлов шифруются при `encryption=on` |

### 3.2 Изоляция ключей

Ключ шифрования загружается в память только при монтировании датасета. При размонтировании ключ выгружается из памяти (`zfs unmount -u`). Дочерние датасеты могут иметь собственные ключи, независимые от родительских.

### 3.3 Совместимость с аппаратным ускорением

На процессорах с поддержкой AES-NI (x86_64) и ARMv8 Crypto Extensions (aarch64) операции шифрования/дешифрования выполняются аппаратно.

## 4. Разграничение доступа

### 4.1 Делегирование прав ZFS

ZFS реализует собственную модель делегирования привилегий, позволяющую предоставлять непривилегированным пользователям конкретные операции с датасетами:

```bash
# Разрешить пользователю создавать снапшоты
zfs allow user1 snapshot tank/data

# Разрешить группе монтировать датасеты
zfs allow -g backup mount,send tank/backup
```

### 4.2 Квоты и ограничения

Квоты предотвращают исчерпание дискового пространства отдельными пользователями или датасетами:

```bash
zfs set quota=100G tank/users/user1
zfs set userquota@user1=10G tank/data
```

### 4.3 Изоляция датасетов

Каждый датасет является независимой единицей файловой системы с собственными правами доступа, квотами и параметрами монтирования.

## 5. Аудит и мониторинг

### 5.1 События пула

```bash
# Просмотр истории операций с пулом
zpool history mypool

# Состояние здоровья пула
zpool status -v mypool
```

### 5.2 Интеграция с системным журналом

ZFS записывает критические события (ошибки чтения/записи, деградация RAID, повреждение данных) в системный журнал (`journald` / `syslog`).

### 5.3 Мониторинг ошибок

```bash
# Количество ошибок по каждому устройству
zpool status -v

# Запуск проверки целостности
zpool scrub mypool
```

## 6. Обновление и управление уязвимостями

Сообщения об уязвимостях принимаются через механизм issue tracker репозитория. Критические уязвимости устраняются в приоритетном порядке с выпуском патч-версий.

История исправлений безопасности: [CHANGELOG.ru.md](CHANGELOG.ru.md)

## 7. Соответствие стандартам

| Стандарт / Требование | Статус |
|-----------------------|--------|
| Шифрование AES-256 | Реализовано (нативное) |
| Контроль целостности данных | Реализовано (SHA-256 / BLAKE3) |
| Разграничение прав доступа | Реализовано (ZFS delegation) |
| Аудит операций | Реализовано (zpool history, syslog) |
| Защищённое хранение ключей | Реализовано (keystore, PKCS#11) |
